# NodIM 即时通讯项目数据流设计（Rust/libp2p 后端 + 前端插件化微内核）

针对你的 NodIM 项目（Rust/libp2p 做消息传输 + 前端插件化微内核），数据流设计的核心是**分层解耦** + **单向数据流** + **插件化状态隔离**，既要适配 libp2p 的 P2P 通信特性，又要满足前端插件化的灵活性。下面从「整体架构」「核心数据流」「插件化数据流」「关键实现细节」四个维度给出完整设计方案。

## 一、先明确核心架构分层（数据流的基础）

先划分清晰的前后端分层，确保数据流方向可追溯、插件可插拔：

前端层Rust 桥接层（Tauri）libp2p 核心层（Rust）微内核 Core业务插件（聊天/联系人/文件）事件转发状态同步权限校验P2P 网络模块消息编解码节点发现数据持久化

```
graph TD
    A[前端层] --> B[Rust 桥接层（Tauri）] --> C[libp2p 核心层（Rust）]
    A --> A1[微内核 Core] & A2[业务插件（聊天/联系人/文件）]
    B --> B1[事件转发] & B2[状态同步] & B3[权限校验]
    C --> C1[P2P 网络模块] & C2[消息编解码] & C3[节点发现] & C4[数据持久化]
```

前端层Rust 桥接层（Tauri）libp2p 核心层（Rust）微内核 Core业务插件（聊天/联系人/文件）事件转发状态同步权限校验P2P 网络模块消息编解码节点发现数据持久化

### 各层核心职责

|     分层      |                           核心职责                           |
| :-----------: | :----------------------------------------------------------: |
|  前端微内核   |    插件管理、全局状态核心、跨插件通信、与 Rust 桥接层交互    |
|   前端插件    | 独立业务逻辑（如聊天窗口、联系人列表）、插件内状态、订阅微内核的全局事件 |
|  Rust 桥接层  | Tauri 命令 / 事件封装、libp2p 事件转发、数据格式转换（Rust ↔ JSON/Protobuf） |
| libp2p 核心层 | P2P 消息收发、节点管理、消息加密 / 解密、本地数据存储（如 SQLite） |

## 二、整体核心数据流（单向闭环）

整体遵循「**前端→桥接层→libp2p**（发送）」「**libp2p→桥接层→前端**（接收）」的单向数据流，核心原则：**状态只读、事件驱动、插件隔离**。

### 1. 消息发送数据流（前端→后端→P2P 网络）

插件 UI 触发（如发送消息按钮）插件内校验（格式/权限）微内核 Core 统一封装Tauri invoke 调用 Rust 桥接函数Rust 桥接层：数据序列化（Protobuf）libp2p 层：加密 + 路由选择（Kademlia/DHT）libp2p 网络：P2P 消息广播/点对点发送本地持久化（Rust 层：SQLite 存储发送记录）Rust 事件通知前端：消息发送成功微内核更新全局状态插件订阅状态更新 → 刷新 UI

```
graph LR
    A[插件 UI 触发（如发送消息按钮）] --> A1[插件内校验（格式/权限）]
    A1 --> B[微内核 Core 统一封装]
    B --> C[Tauri invoke 调用 Rust 桥接函数]
    C --> D[Rust 桥接层：数据序列化（Protobuf）]
    D --> E[libp2p 层：加密 + 路由选择（Kademlia/DHT）]
    E --> F[libp2p 网络：P2P 消息广播/点对点发送]
    F --> G[本地持久化（Rust 层：SQLite 存储发送记录）]
    G --> H[Rust 事件通知前端：消息发送成功]
    H --> I[微内核更新全局状态]
    I --> J[插件订阅状态更新 → 刷新 UI]
```

插件 UI 触发（如发送消息按钮）插件内校验（格式/权限）微内核 Core 统一封装Tauri invoke 调用 Rust 桥接函数Rust 桥接层：数据序列化（Protobuf）libp2p 层：加密 + 路由选择（Kademlia/DHT）libp2p 网络：P2P 消息广播/点对点发送本地持久化（Rust 层：SQLite 存储发送记录）Rust 事件通知前端：消息发送成功微内核更新全局状态插件订阅状态更新 → 刷新 UI

#### 关键步骤说明

-   **前端侧**：插件不直接调用 Tauri 接口，而是通过微内核的「消息发送 API」封装，微内核统一处理「重复发送」「离线缓存」「进度回调」；
-   **Rust 桥接层**：将前端的 JSON 数据转为 Rust 结构体，再交给 libp2p 层处理，同时做「消息 ID 生成」「加密（如 ChaCha20）」；
-   **libp2p 层**：根据消息类型（点对点 / 群聊）选择传输方式，点对点用 `libp2p-ping`/`libp2p-identify`，群聊用 `libp2p-gossipsub`；
-   **状态同步**：发送成功后，Rust 层通过 Tauri `emit` 触发前端事件，微内核更新全局「消息列表」状态，插件订阅后刷新 UI。

### 2. 消息接收数据流（P2P 网络→后端→前端）

libp2p 节点接收消息Rust 层：解密 + 验签 + 解析Rust 层：本地持久化（未读/已读标记）Rust 桥接层：事件转发（Tauri emit）前端微内核：接收事件并校验（插件权限）微内核：更新全局状态（如新增消息）插件订阅对应状态 → 更新 UI（如聊天窗口、未读红点）插件触发本地逻辑（如消息提醒、声音）

```
graph LR
    A[libp2p 节点接收消息] --> A1[Rust 层：解密 + 验签 + 解析]
    A1 --> B[Rust 层：本地持久化（未读/已读标记）]
    B --> C[Rust 桥接层：事件转发（Tauri emit）]
    C --> D[前端微内核：接收事件并校验（插件权限）]
    D --> E[微内核：更新全局状态（如新增消息）]
    E --> F[插件订阅对应状态 → 更新 UI（如聊天窗口、未读红点）]
    F --> G[插件触发本地逻辑（如消息提醒、声音）]
```

libp2p 节点接收消息Rust 层：解密 + 验签 + 解析Rust 层：本地持久化（未读/已读标记）Rust 桥接层：事件转发（Tauri emit）前端微内核：接收事件并校验（插件权限）微内核：更新全局状态（如新增消息）插件订阅对应状态 → 更新 UI（如聊天窗口、未读红点）插件触发本地逻辑（如消息提醒、声音）

#### 关键步骤说明

-   **libp2p 层**：接收消息后先验证节点身份（libp2p 公钥），确保消息来源合法；
-   **Rust 桥接层**：将 Rust 结构体转为 JSON，通过 Tauri `emit("message-received", 数据)` 推送给前端；
-   **前端微内核**：统一接收所有后端事件，根据「消息类型 + 插件注册的处理器」分发到对应插件，避免插件直接监听后端事件导致耦合；
-   **插件侧**：仅订阅自己关心的状态（如「聊天 - 张三」的消息），不处理全局状态，保证隔离性。

## 三、前端插件化微内核的数据流设计（核心重点）

前端采用「**微内核 + 插件**」架构，微内核负责「核心能力 + 数据流调度」，插件负责「业务逻辑 + UI」，数据流设计需满足：插件无感知后端、插件间低耦合、状态可隔离。

### 1. 微内核（Core）的核心能力（数据流中枢）

微内核只保留最核心的数据流能力，不包含任何业务逻辑：

|  核心模块  |                          数据流职责                          |
| :--------: | :----------------------------------------------------------: |
| 插件管理器 |        插件注册 / 卸载、插件权限校验、插件间通信转发         |
| 状态管理器 | 全局状态（如当前登录节点、在线状态）存储，插件状态隔离存储（Pinia 模块化） |
|  事件总线  |  统一接收 Rust 后端事件、转发给对应插件；处理插件间事件通信  |
| 桥接适配器 | 封装 Tauri invoke/emit 接口，统一处理前后端数据转换、错误兜底 |
| 数据持久化 | 前端本地缓存（如插件配置、未同步消息），与 Rust 层存储做双向同步 |

### 2. 插件的数据流设计（隔离 + 复用）

每个插件是独立的模块，通过「微内核 API」与外部交互，不直接操作全局状态 / 后端接口：

#### 插件内部数据流（单向）

```plaintext
插件 UI → 插件 action → 微内核 API → 桥接层 → 后端
后端 → 桥接层 → 微内核事件 → 插件 mutation → 插件 state → 插件 UI
```

#### 插件类型与数据流隔离

| 插件类型 |       示例        |                        数据流隔离方式                        |
| :------: | :---------------: | :----------------------------------------------------------: |
| 核心插件 |  登录 / 节点管理  |                 访问全局状态，可触发全局事件                 |
| 业务插件 | 聊天窗口 / 联系人 | 仅访问自己的命名空间状态（如 `chat-zsan`），通过微内核订阅全局事件（如「节点上线」） |
| 工具插件 |  表情 / 文件传输  |    无独立状态，通过微内核提供「工具 API」，由业务插件调用    |

### 3. 前端状态管理设计（Pinia 模块化）

基于 Pinia 实现「全局核心状态 + 插件隔离状态」，避免状态混乱：



```typescript
// src/core/store/index.ts（微内核全局状态）
import { defineStore } from 'pinia';
export const useCoreStore = defineStore('core', {
  state: () => ({
    // 全局核心状态：P2P 节点信息、在线状态、已加载插件
    nodeId: '', // libp2p 节点 ID
    isOnline: false,
    loadedPlugins: [],
    globalConfig: {},
  }),
  actions: {
    // 微内核统一的状态更新方法，插件不可直接修改
    updateNodeStatus(isOnline: boolean) {
      this.isOnline = isOnline;
      // 触发全局事件，通知所有插件
      this.$emit('node-status-changed', isOnline);
    },
  },
});

// src/plugins/chat/store.ts（插件隔离状态）
import { defineStore } from 'pinia';
// 插件状态命名空间：插件名 + 唯一标识
export const useChatStore = defineStore('plugin-chat', {
  state: () => ({
    // 仅属于聊天插件的状态：消息列表、未读数、当前会话
    sessions: {},
    unreadCount: 0,
    currentSessionId: '',
  }),
  actions: {
    // 插件内部状态更新，仅响应微内核转发的事件
    addMessage(message: Message) {
      this.sessions[message.sessionId].messages.push(message);
      this.unreadCount++;
    },
  },
});
```

## 四、关键细节：适配 libp2p 的特殊数据流处理

### 1. 离线消息处理

libp2p 节点离线时，消息需缓存并在重连后自动发送：

插件发送消息微内核检测节点离线前端本地缓存（IndexedDB）微内核订阅「节点上线」事件节点上线后，微内核批量调用 Rust 桥接层Rust 层发送缓存消息发送成功后，删除前端/后端缓存

```
graph LR
    A[插件发送消息] --> A1[微内核检测节点离线]
    A1 --> B[前端本地缓存（IndexedDB）]
    B --> C[微内核订阅「节点上线」事件]
    C --> D[节点上线后，微内核批量调用 Rust 桥接层]
    D --> E[Rust 层发送缓存消息]
    E --> F[发送成功后，删除前端/后端缓存]
```

插件发送消息微内核检测节点离线前端本地缓存（IndexedDB）微内核订阅「节点上线」事件节点上线后，微内核批量调用 Rust 桥接层Rust 层发送缓存消息发送成功后，删除前端/后端缓存

### 2. 消息一致性校验

P2P 网络消息可能乱序 / 重复，需在 Rust 层做幂等处理：

-   每条消息生成唯一 ID（如 `节点ID + 时间戳 + 随机数`）；
-   Rust 层维护「已处理消息 ID 列表」，重复消息直接丢弃；
-   前端接收消息时，通过消息 ID 去重，保证 UI 展示一致。

### 3. 插件化的消息扩展

支持插件自定义消息类型（如文件、表情、语音），数据流需兼容扩展：



```typescript
// 微内核定义消息扩展接口
export interface MessageExtension {
  type: string; // 插件自定义类型，如 "plugin-file"
  handler: (data: any) => Promise<any>; // 插件处理函数
}

// 文件插件注册消息扩展
core.registerMessageExtension({
  type: "plugin-file",
  handler: async (data) => {
    // 插件处理文件上传/加密
    const encryptedData = await encryptFile(data.file);
    return { ...data, encryptedData };
  },
});

// 微内核发送时自动调用扩展处理器
async function sendMessage(message: Message) {
  if (message.extensionType) {
    const extension = core.getMessageExtension(message.extensionType);
    message.data = await extension.handler(message.data);
  }
  await tauri.invoke('libp2p_send_message', { message });
}
```

## 五、数据流设计的核心原则（总结）

1.  **单向数据流**：前端→桥接层→libp2p /libp2p→桥接层→前端，禁止反向 / 双向修改状态，保证可追溯；
2.  **插件隔离**：插件仅通过微内核交互，不直接访问后端 / 其他插件状态，降低耦合；
3.  **状态分层**：全局核心状态（微内核）+ 插件隔离状态（业务），避免状态混乱；
4.  **事件驱动**：所有状态更新由事件触发（后端事件 / 插件事件），无直接状态修改；
5.  **适配 P2P**：兼容 libp2p 离线、乱序、节点动态变化的特性，做好缓存 / 幂等 / 重连处理。

### 最终数据流核心链路回顾

1.  **发送**：插件 UI → 微内核封装 → Tauri 调用 → Rust 序列化 → libp2p 发送 → 本地存储 → 前端状态更新；
2.  **接收**：libp2p 接收 → Rust 解密解析 → Tauri 事件 → 微内核分发 → 插件状态更新 → UI 刷新；
3.  **插件化**：微内核作为中枢，统一管理插件注册、事件转发、状态调度，适配 P2P 网络特性。
